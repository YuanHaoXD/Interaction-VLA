## 一、先破除最大的误会:模板动作 ≠ 模型将来的动作库

### 你的困惑(完全合理)

> "nod、摇天线这些动作都是提前编码好的。可是我要做的是像 VLA 那样,让模型**自主输出连续的空间动作序列**,而不是从一个硬编码动作库里挑。为什么演示里还在用这些编码好的动作?"

### 一句话回答

**你看到的回放,是在播放一条"合成出来的训练数据",不是模型的输出——因为现在还没有模型(M1 阶段只搭地基、不训练)。`nod` 这些模板不是"模型的动作词汇表",而是"造训练数据的临时工具/替身演员"。**

### 关键:存进文件的,是纯连续数字,里面根本没有"nod"

每一段数据的核心文件是 `actions.npy`。我们打开真实的一段(`samples/ep_00000003`)看看它到底是什么:

```
形状: (719, 9)   dtype: float32
维度顺序: [x, y, z, roll, pitch, yaw, body_yaw, ant_右, ant_左]
```

它就是一个**表格**:719 行(23.96 秒 × 30 帧/秒),每行 9 个小数,代表那一瞬间头部平移、头部转动、身体转动、两根天线各自转到哪。**这就是 VLA 要输出的东西——连续空间动作序列**,一模一样。

这段数据里有个"摇头"事件(标签写在 meta 里,标注是 8.3 秒)。我们把摇头最猛那一刻附近的 `yaw`(摇头对应的维度)逐帧打印出来:

```
帧278 (t=9.27s): yaw=-0.1389
帧279 (t=9.30s): yaw=-0.1547
帧280 (t=9.33s): yaw=-0.1593   ← 峰值
帧281 (t=9.37s): yaw=-0.1528
帧282 (t=9.40s): yaw=-0.1359
帧283 (t=9.43s): yaw=-0.1102
```

看到没?这是一条**平滑变化的连续曲线**(一个衰减正弦)。整段里 `yaw` 有 **350 个互不相同的浮点数**。如果"摇头"真是个硬编码开关,那只会有"开/关"两三个值。**它是连续的。文件里没有任何地方写着"nod"或"shake_head"这个词——那只是我们生成这串数字时用的配方名。**

### 那"编码好的模板"到底扮演什么角色?

三层关系,理清就通透了:

**① 执行层(现在和将来完全一样):** 无论数字从哪来,最终都是"每 1/30 秒把 9 个数推给机器人"。你看到的回放,机器人**已经是被连续 9 维序列驱动的**,不是被某个 `nod()` 函数驱动的。这半边 B1–B4 已经建好、验证过,换模型时一行都不用改。

**② 数据来源层(这里现在用模板,将来用模型):**

```
现在(M1 地基):   离散标签"nod" ──[模板生成器]──▶ [719×9]连续轨迹
将来(M2 之后):   对话+画面    ──[JoyAI慢脑+动作专家快脑]──▶ [T×9]连续轨迹
                                                        ▲
                                       换的只是"数字从哪来",右边执行层不变
```

**③ 为什么第一步非得用模板?** 因为我们**没有现成的"机器人动作录像"可以训练**。手上有的,是另一条工作线(action-annotation)用 Gemini 给 300 万条对话打的**离散标签**:"这一刻,合适的动作类别是 nod / shake_head / …"。

要训练一个**连续**模型,得给这些**离散标签**配上**连续的目标轨迹**当"标准答案"。模板生成器干的就是这件事:把"nod"这个词,廉价地展开成一条**参数随机化**的连续曲线。这是设计文档 §4.1 说的**第 1 层数据(程序化合成,百万级)**。

### 那模型会不会就把这 5 个模板背下来?——不会,而且专门防了

这是你担心的核心,也是设计里最讲究的地方:

1. **每条轨迹都不一样(参数随机化 + 叠加)。** 每次生成 nod,幅度、频率、周期数、衰减、相位、起始时刻抖动**全是随机**的,还叠加了贯穿全程的 idle 微动、用户说话时的 backchannel 点头。模型**永远看不到两条相同的轨迹**。它学到的是一个**连续分布(流形)**,不是 5 个固定点。回归 / 流匹配模型会在这个流形上**插值、混合、外推**——能输出训练集里逐字没有的动作(比如"点头正过渡到歪头")。

2. **第 2 层:真人手把手示教(设计文档 §4.2)。** 中后期,学长/你会**断掉电机、亲手掰着机器人**,对着真实对话演"身体语言",30Hz 录下真实的连续轨迹。这里面有模板永远写不出的韵律微结构(重音时一顿、犹豫时慢慢歪头)。模型**最终在这层真人数据上精调**。

3. **测试只用真人数据,和模板零重叠(设计文档 §4.4 防泄漏)。** 评测集只来自第 2 层(真人掰的),模板参数分布和它**没有交集**。所以我们是在**如实测量"模型有没有泛化到模板之外"**,不是自己骗自己。

### 最后一击:你担心的"硬编码动作库",正是我们要打败的对手

`reachy-mini-demo` 里现在跑的"小艺",就是那种老做法:模型调用一个写死的 `nod()` 函数——**离散、生硬、上一个动作没做完就被下一个打断**。那是设计文档 §5.3 里明确列的**系统基线(baseline),是我们要超越的对象**。

**整个 interaction-vla 项目的贡献,就是用"连续生成"取代"硬编码动作库"。** 所以你的直觉和项目目标**完全一致**——你只是在"造数据"这一步撞见了脚手架,主角(模型)还没登场。

## 二、B1–B4 到底在干什么 + 代码怎么和仿真交互

### 大局:整个系统是"两个脑"(设计文档 §3.1)

- **慢脑**(JoyAI 8B,每秒决策 1 次,冻结不训):看对话和摄像头,决定"说什么 / 要不要沉默",吐出一个**语义隐状态**(一串代表当前意图的数字)。
- **快脑 = 动作专家**(50–200M 小模型,每秒 30 次):拿慢脑的隐状态 + 机器人当前实际姿态,输出**未来 1.5 秒的连续动作块**(45 帧 × 9 维)。
- 中间靠 **action chunking** 桥接两个不同的节奏(1Hz 决策 → 30Hz 控制),块与块重叠处加权混合,天然连续无跳变(设计文档 §2 洞察 1)。

**M1 阶段这两个脑都还没造。** M1 只造**地基**:动作用什么格式表示、训练数据怎么批量造、仿真怎么把数据播出来给人看、怎么自动校验数据合不合格。B1–B4 就是这套地基。

### 四步各自在干嘛(大白话)

| 步骤            | 大白话比喻         | 具体做了什么                                                 | 产出文件                                         |
| ------------- | ------------- | ------------------------------------------------------ | -------------------------------------------- |
| **B1 动作空间勘察** | 量画布尺寸         | 让机器人 9 个维度逐个动到极限,量出"每维安全能动多大";再用正弦波压测"30Hz 推得动吗"       | `sim/limits.json`                            |
| **B2 轨迹模板库**  | 造数据合成器        | 写 5 类标签的"随机连续轨迹生成器" + 全程 idle 微动 + 说话时 backchannel 点头  | `data_gen/templates.py`                      |
| **B3 格式+播放器** | 定文件规范 + 录像机   | 定义一段数据的**标准目录格式**和**自动校验器**;写一个**回放器**把存好的轨迹送进仿真       | `data_gen/schema.py`、`sim/replay_episode.py` |
| **B4 端到端样例**  | 试产 100 件 + 质检 | 把 B1/B2/B3 串起来批量造 100 段,抽 3 段肉眼看质量(有没有呼吸感、动作明不明显、流不流畅) | `samples/`(100 段)                            |

**为什么是这个顺序:** 先量画布(B1)→ 才知道笔能画多大(B2 生成轨迹要限制在安全范围内)→ 有了轨迹要定存储格式和播放器(B3)→ 最后拼起来批量产 + 验收(B4)。每一步都依赖上一步的产出。

### 一段数据长什么样(B3 定的标准格式,契约 C0)

```
samples/ep_00000003/
  meta.json       # 元信息:时长、有哪些事件(标签+时刻)、随机种子…
  timeline.json   # 对话时间线:几秒时用户说话、几秒时机器人回复(带动作标签)
  actions.npy     # 核心:[719, 9] 的连续动作序列,30Hz。← 就是上面讲的那串数字
```

`meta.json` 里 `template_events` 记着 `[{label:"shake_head", t:8.3}, {label:"nod", t:15.4}]`——**这是"生成配方"的存根,给将来训练时对时间线用的**,不是动作本身。动作本身在 `actions.npy` 里,是连续数字。

### 代码怎么和仿真"说话"(交互模式)

**角色:**

- `reachy-mini-daemon --sim --scene minimal` 这个进程 = **虚拟机器人本体**。它内部跑 MuJoCo 物理引擎(算重力、关节、碰撞),同时在 `localhost:8000` 开一个服务端口等人来控制。你看到的 3D 窗口就是它渲染的。
- 我们的 Python 脚本用 `reachy_mini` 这个 SDK 连上它,当**客户端**发指令。

**核心控制循环(30Hz):**

```
每 1/30 秒重复:
  1. 拿到这一帧的 9 个数 [x,y,z,roll,pitch,yaw,body_yaw,ant右,ant左]
  2. 把前 6 个组装成一个 4×4 姿态矩阵(SDK 的标准头部姿态格式)
  3. mini.set_target(head=矩阵, antennas=[右,左], body_yaw=φ)  ← 指令发给 daemon
  4. daemon 收到 → 做 Stewart 平台逆解(算 6 根连杆各伸多长)→ MuJoCo 步进一帧
  5. 3D 窗口里机器人动一下
  6. 忙等到下一个 1/30 秒时刻,回到 1
```

**反向读取:** `mini.get_current_head_pose()` 能读回机器人**实际达到**的姿态。

- B1 用它测限位:发一个指令,读回实际到了哪,误差太大就说明"顶到极限了"。
- 将来它是**快脑的本体反馈通道**——让模型永远"知道身体现在在哪",即使被人手碰了也能自然接续(设计文档 §3.5)。

**`set_target` vs `goto_target`(一个重要区别):**

- `set_target`:**直接推一帧,不插值**。高频控制(30Hz 喂轨迹)用这个——每帧姿态是我们自己算好的。**我们全程用它。**
- `goto_target`:带平滑插值,你给一个目标+时长,它自己缓慢过渡过去。适合"过去 2 秒缓缓转到某个方向"这种低频指令。硬编码版小艺的 `nod()` 就是用这个——**这也是它生硬的原因之一**。

### 现在 vs 训练出模型后(这张图最能解你的惑)

```
现在 (M1 地基):
   离散标签"nod"
        │
        ▼
   [模板生成器]  ← data_gen/templates.py (B2)
        │
        ▼
   [719×9]连续轨迹 存成 actions.npy  ← data_gen/schema.py (B3)
        │
        ▼
   set_target @30Hz  ← sim/replay_episode.py (B3)
        │
        ▼
   MuJoCo 仿真 (你看到的回放)


将来 (M2 之后,训练出模型):
   对话上下文 + 摄像头帧
        │
        ▼
   [JoyAI 8B 慢脑]  →  语义隐状态
        │
        ▼
   [动作专家 快脑]   ← 这才是要训练的模型
        │
        ▼
   [T×9]连续动作块
        │
        ▼
   [安全层]  ← 硬限幅,防撞(用 B1 的 limits.json)
        │
        ▼
   set_target @30Hz  ←──────┐
        │                    │ 和现在完全相同的执行层

```

**核心洞察:那条"`[T×9] → set_target@30Hz → 仿真`"的执行轨道,B1–B4 已经铺好、跑通、验证过了。** 现在用模板当"临时列车"证明轨道能跑;M2 训练出模型后,把"真列车"(动作专家)开上同一条轨道即可。**这就是为什么地基阶段先用模板——先修路验路,再上车。**

## 三、如果你只记三句话

1. **`nod`/`wiggle` 是"造数据的配方名",不是"模型的动作库"。** 存进文件的是连续浮点数(摇头 = 350 个不同的 yaw 值组成的平滑曲线),和 VLA 要的连续序列一模一样。现在还没有模型,你看到的是训练数据回放。

2. **B1–B4 在铺"执行轨道"**:量安全范围(B1)→ 造数据合成器(B2)→ 定格式+播放器(B3)→ 批量产+质检(B4)。代码通过 `set_target`@30Hz 驱动 `reachy-mini-daemon` 里的 MuJoCo 虚拟机器人。

3. **硬编码动作库(现在的小艺)正是我们要打败的基线。** 项目全部意义就是用"模型自主生成连续动作"取代它——你的直觉从头到尾都是对的。
