## 0. 一句话与目标

**在开源实时交互模型 JoyAI-VL-Interaction 上外挂一个连续动作专家，让 Reachy Mini 在全双工交互中（说话时、聆听时、沉默时）流畅地输出拟人动作，并发布首个"交互中机器人动作"的开放基准。**

- 目标会议：**CVPR 2027**（约 2026-11 中旬截稿）。ICLR 2027（2026-09 底）仅在 9 月结果已惊艳时顺路评估，不为它分心。
- 影响力策略：论文 + 代码 + 模型 + 数据 + 基准**全开源**（公司已许可），对齐 JoyAI 自身的全开放打法。
- 团队：用户（实习生，VLM 背景，VLA 新手）+ 学长（实质参与：录数据、把关故事）+ 本地 Fable 5（设计/审查）+ 服务器 Opus 4.8（实施/实验）。
- 算力：8×昇腾 910B 64G 调试 + 集群训练。**注意：8B 模型微调链路尚未在该环境跑通**（候选工具：LLaMA-Factory / ms-swift）。主线只训小动作专家（纯 PyTorch，A1 冒烟覆盖），8B 微调仅 v3（B 收尾）需要，见 §7 风险 3。

## 1. 相关工作定位（查重结论，2026-07-14）

| 工作                                       | 做了什么                                                  | 缺什么（=我们的差异点）                 |
| ---------------------------------------- | ----------------------------------------------------- | ---------------------------- |
| RoboEgo / FLM-Ego（BAAI，arXiv:2506.01934） | 全双工全模态+人形机器人，边说边动                                     | 动作是**离散 token**；闭源；无基准       |
| DyaPlex（arXiv:2606.03874，2026-06）        | 全双工语音+动作双塔（冻结语音基座+动作通道），Seamless Interaction 4000h 训练 | 纯语音输入、**虚拟人**动作、无视觉、无主动性、无实体 |
| LiveGesture（arXiv:2604.10927）            | 流式零前瞻 co-speech 全身手势                                  | 虚拟人、纯音频驱动、无理解-决策环节           |
| ELEGNT（Apple ML Research）                | 台灯机器人表达性动作设计研究                                        | 非学习模型（动机引用）                  |
| Reachy Mini 情绪感知研究（arXiv:2605.12786）     | 证明该平台低自由度表达"社会上有效"                                    | （平台合法性引用）                    |
| HuggingFace Reachy Mini Conversation App | 语音对话+规则动作库                                            | 规则驱动非学习（**系统基线**）            |

**空位声明**：视觉为核心、自主决定何时开口的交互模型 + 连续表达性动作（说话+聆听）+ 实体开源机器人 + 开放基准——四要素合体的位置无人占据。DyaPlex 2026-06 出现说明方向正热，窗口在收窄，节奏必须快。

**通用性主张（写进论文）**：动作专家与交互模型之间的接口刻意收窄为"低频语义隐状态（可降级为回复文本）+ 本体反馈"。任何未来全双工模型（MiniCPM-o、Qwen-Omni、DuplexOmni 开源后）都可替换慢脑而小脑不动；条件接口预留扩展位——当基座原生具备音频模态时，可并入音频特征通道实现音节级韵律跟随。本工作是框架，不是对某个模型的 hack。

## 2. 三个核心洞察（驱动全部设计决策）

1. **节奏差**：JoyAI 每秒决策一次（1Hz），机器人控制需 50Hz。桥接 = **action chunking**：每个决策步输出未来 1.5–2s 的动作块，块间重叠区时间加权混合（temporal ensembling），天然连续无跳变。这同时消解了离散动作 token"上一动作未完成即被下一 token 冲突"的问题。
2. **沉默也是数据**：JoyAI 训练格式为每秒一对 user/assistant（`</silence>` 或 `</response> 文本`）。沉默秒同样可带动作监督（聆听点头、思考歪头）。**"沉默时也在动"是回合制 VLA 结构上做不到的**——故事的核心新颖点。
3. **无音频原则（2026-07-14 定）**：JoyAI 本质上没有语音模态——ASR/TTS 都是外挂，8B 模型从未见过声音波形。给动作专家接音频等于"小脑听得见、大脑听不见"的系统级缝合，不是模型能力。因此**动作专家初期不接任何音频通道**，目标收紧为：**图像/文本输入 → 对应且流畅的动作**；前期实验也不接语音（纯文本进出 + 动作）。发声区间用**时长代理**（字数÷语速估算，把语义条件摊开在区间上）；聆听 backchannel 由隐状态/时间线中的 user_text 事件驱动（模型知道"用户在说话"，时机精度秒级）。音节级韵律跟随（重音点头）明确写为 limitation + future work：待优质开源全双工全模态基座出现、慢脑原生具备音频模态时，在动作专家预留的条件接口上并入音频通道。

## 3. 架构设计

### 3.1 总览（慢脑 + 快脑）

```
用户文本(实验期直接输入;部署期由ASR产生) ─┐
摄像头帧(≈1fps) ─────────────────────────┼─▶ JoyAI 8B（慢脑,1Hz,冻结）─▶ 文本回复(部署期→TTS)
                                         │        │ 决策token隐状态(训练期离线预计算)
                                         │        ▼
                                 动作专家（快脑,50–200M,50Hz） ─▶ 动作块 ─▶ 安全层 ─▶ set_target
                                     ▲                                        ▲
机器人实测状态流 ────────────────────┘                         对视伺服(叠加) ──┘
※ 音频不进任何模型:动作专家条件接口预留扩展位,待全模态基座时并入音频通道
```

### 3.2 动作表示（项目的"字母表"）

任务空间 9 维向量，**50Hz**（v1.3 起；原 30Hz），逐维归一化到 [-1,1]（统计范围来自数据集）。50Hz 决策理由：官方动作库原生 50Hz（降采样=对最高质量资产有损）、遥操作金标准将照 50Hz 录制、合成模板是参数化函数采样率无关、`set_target` 实测 0.18ms/call 支撑充裕；迁移趁大规模数据生成前完成（M1.5）。

```
a_t = [x, y, z,                # 头部平移（米，世界系）
       roll, pitch, yaw,       # 头部姿态（弧度，世界系；SDK 的 Stewart 逆解自动补偿 body_yaw）
       body_yaw,               # 身体旋转（弧度）
       ant_R, ant_L]           # 双天线（弧度）
```

- 执行接口：`reachy_mini.set_target(head=4×4矩阵, antennas=[右,左], body_yaw=φ)`，30Hz 流式调用；4×4 矩阵由 (x,y,z,rpy) 组装。官方开发者文档：https://reachymini.net/zh/developers.html （电机/SDK 适配细节以此+SDK 源码为准，M1 服务器侧任务）。
- 状态读取：`get_current_head_pose()`（4×4）+ `get_current_joint_positions()`，同样 30Hz 采样，供数据录制与反馈通道。
- 选任务空间不选关节空间的理由：SDK 原生接口即任务空间（仿真/真机零适配）、可解释（pitch 正弦=点头）、跨机型可重定向。
- 旋转表示注记：头部运动范围小（±30° 量级，远离万向锁），rpy 够用；若后期出现插值伪影，切换 6D 连续旋转表示（Zhou et al. 2019），接口不变。

### 3.3 动作专家（快脑）

**输入条件通道**（对齐到统一的 100ms 内部时间步；"阶段"列 = 何时并入）：

| 通道           | 内容                                                      | 频率   | 阶段              | 备注                                                                                           |
| ------------ | ------------------------------------------------------- | ---- | --------------- | -------------------------------------------------------------------------------------------- |
| 语义隐状态        | JoyAI 决策 token 处 last-layer 隐状态，线性投影；发声区间用"字数÷语速"时长代理摊开 | 1Hz  | **v1**          | "好的"→点头的语义来源；携带视觉上下文与开口前意图（沉默时对视觉事件的反应靠它）；聆听 backchannel 也由它驱动（隐状态知道用户在说话）；"交互模型原生驱动身体"主张的载体 |
| 本体           | 机器人**实测**状态流（非模型自身输出）                                   | 50Hz | **v1**          | 见 3.5 反馈设计                                                                                   |
| 回复文本（Plan B） | JoyAI 回复文本 → 小型可训练文本编码器                                 | 事件驱动 | 备用              | **仅当隐状态导出受阻时**替代语义通道：纯 API 级联、JoyAI 零改动，代价是丢失视觉上下文与沉默秒语义                                     |
| 音频（预留扩展位）    | 用户/自身语音特征                                               | —    | **future work** | 无音频原则（§2 洞察 3）：待全模态基座出现再并入，实现音节级韵律跟随                                                         |

> 隐状态获取不依赖 vLLM 改造：**训练期离线预计算**（transformers 路径跑一遍数据集并缓存）；M2 仿真 demo 为 1Hz 决策，8B 走 transformers 慢路径每秒一次前向在 910B 上可行；在线 vLLM 导出留到真机 demo 前解决。

**输出**：动作块 = 未来 1.5s @ 50Hz = 75 步 × 9 维（块长/重叠为可调超参，初始 1.5s 块 / 0.5s 重叠）。

**两版实现**（先后关系，不是二选一）：

- **v1（8 月，先跑通）**：因果 transformer 编码 v1 二通道（语义隐状态+本体；隐状态训练期离线预计算）→ MLP 头直接回归整块。损失 = SmoothL1 + 一阶差分（速度）+ 二阶差分（加速度）正则。目的：让全链路活起来，指标有底线值。若隐状态导出首周受阻 → 切换回复文本 Plan B（见 §3.3 表），接口不变。
- **v2（9 月，论文主结果）**：编码器 → **DiT 风格流匹配解码器**，以上下文向量为条件从噪声去噪出整块。理由：动作分布多峰（同一句话可点头可歪头），回归会把峰平均成"木头动作"，flow matching 是标准解法（π0 同款）。
- **v3 / "B 收尾"（10 月，可选实验）**：架构不变，解冻 JoyAI（LoRA），慢脑+快脑联合微调。涨点 → 主结果升级"端到端"；不涨 → 消融结论"冻结基座已足够"（DyaPlex 同款结论），横竖有产出。

### 3.4 与 StarVLA / 生态的关系

- **StarVLA**（arXiv:2604.05014，乐高化 VLM→VLA 框架）：**当零件库不当整机**。直接借：流匹配动作头实现、训练 loop、动作归一化工具。不可整机用：它面向"图像+指令→操作"标准 VLA，我们的条件流（1Hz 隐状态+对话时间线）非标，数据管线自写。
- **LeRobot 数据格式**：episode 存储采用 LeRobotDataset 风格（parquet+媒体文件），白嫖其可视化与生态兼容。
- 两者的 torch_npu 兼容性列入 M1 冒烟测试（见 §7 风险）。

### 3.5 反馈、安全与仲裁

- **闭环反馈**：下一块的生成以机器人**实测状态**为历史条件（而非模型上一块输出）。因为对视伺服叠加在模型输出之上，实际姿态≠模型输出；喂实测状态使模型永远"知道身体在哪"，外部扰动（人手碰机器人）后能自然接续。训练时对该通道加高斯噪声增广，防暴露偏差。
- **安全层**（模型与硬件之间的硬包裹，不可绕过）：姿态限幅（工作空间盒）、角速度/角加速度上限、天线限位；违例时按最近可行值裁剪并记日志（违例率本身是评测指标）。
- **对视伺服**：人脸跟随沿用 reachy-mini-demo 的成熟视觉伺服（时间常数型增益），作为低层**叠加**在模型输出上（yaw/pitch 小幅修正）。v1 不让模型学对视——对视是控制问题，表达是生成问题；后期可并入学习目标（开放点）。

## 4. 数据设计（三层配方）

### 4.1 第 1 层：程序化合成（立即做，百万级，支撑全部前期训练）

**v1.3 重构：动作词汇的核心来源升级为官方录制动作库**（资产 LIB-01，`data_gen/motion_library/`：104 个官方动作 = 85 情绪 + 19 舞蹈，源自 pollen-robotics 官方 HF 数据集，apache-2.0，原生 50Hz；**每个动作带"神态/适用情况"英文描述**，84 个带配套语音）。职业动画师手笔，质量与丰富度远超手写正弦。

- **事件动作（主力）**：标注任务 v2（见下）为每个 response/情境从官方词表选一个动作名（或 none）→ 取库中原始轨迹做**扰动增广**（时间缩放、幅度缩放、起点抖动）后叠加进 episode。
- **手写正弦模板降为补充词汇**：nod/shake_head/tilt_head/wiggle_antennas 等库中较弱的基础对话动作仍用参数化模板；**idle 微动与聆听 backchannel 两类"非事件"动作不变**（程序生成，拟人感的关键）。
- **标注任务 v2**：改造 action-annotation 流水线——prompt 附上 85 个官方动作的 description，让标注模型从中选最贴切者（选不出则 none），替代原 5 类标签。旧 5 类标注（若已产出）走手写模板路径，两路数据可混合。
- 时间锚定：动作起点相对 `</response>`/触发时刻加 -0.2s~+0.5s 抖动；避免机械准时。
- 防泄漏补充：测试集金标准仍只来自第 2 层示教（§4.4 不变）；官方库轨迹入训练不构成循环论证（与金标准不同源）。可选：留 5–10 个官方动作 held-out 做"未见动作泛化"探索实验。
- 产出：episode = (对话时间线上下文, 50Hz×9 维轨迹)，十万级起步、可扩百万级。

### 4.2 第 2 层：手把手示教（中后期，几百段，金标准）

- 方法：真机 `disable_motors()` 断力矩 → 学长/用户**对着真实对话**用手直接掰机器人演"身体"，`get_current_head_pose()` 等以 30Hz 录 9 维状态流 + 对话音频时间线。仿真回放校验后入库。
- 用途：**测试集金标准 + 精调数据**。此层不可省略，因为：① 基准金标准若与训练同源（同一轨迹生成器），评测即循环论证，审稿必杀；② 人手轨迹带模板写不出的韵律微结构（重音一顿、犹豫慢歪头），是 human eval 取胜的关键。
- 录制必须对着真实对话进行（不能凭空摆），否则节奏感丢失。

### 4.3 第 3 层：人类视频重定向（可选，时间富余再做）

Seamless Interaction（Meta 4000h 双人对话）→ 6D 头姿态估计 → 线性映射到 Reachy 头空间；天线以情绪强度规则合成。做了则论文获得"数千小时人类对话重定向"的规模故事；不做不伤主线。**默认不做，M3 末视进度决定。**

### 4.4 训练课程与防泄漏

- 课程：第 1 层预训练 →（第 3 层若做则混入）→ 第 2 层精调。
- 防泄漏：测试集只来自第 2 层（人工掰的）；第 1 层模板参数分布与任何评测数据无交集。

## 5. 评测设计（基准本身就是贡献）

三层金字塔，从便宜到贵：

### 5.1 客观轨迹指标（自动，每次实验必跑）

在第 2 层金标准测试集上：

- **FRMD**（Fréchet Robot Motion Distance）：FGD 移植到 9 维动作空间，轨迹自编码器提特征后算 Fréchet 距离；
- **时机对齐分**：动作起始/峰值 vs 时间线事件（response/user_text 时刻）的对齐度（无音频原则下的秒级替代；音节级节拍对齐留给 future work）；
- **多样性**：同一上下文多次采样的轨迹间平均距离（专治回归"木头病"）；
- **物理可执行性**：jerk 均值、安全层违例率；
- **行为正确率**：利用 `</silence>`/`</response>` 结构可判——聆听段有无 backchannel、否定语义处有无摇头等。

### 5.2 VLM 裁判（半自动，适切性规模化）

生成动作在 MuJoCo 渲染成短视频，连同对话上下文喂强 VLM 打分（语义适切、时机）。

- 成本兜底：只评**精选评测集**（数百段）；备选廉价方案 = 轨迹曲线图+关键帧拼图喂图像 VLM（成本降一个量级）；两方案都用人工评分小集校准裁判可信度后择优。
- 协议（prompt、渲染参数、打分表）随基准开源，保证可复现。

### 5.3 人类评测（投稿前一次）

并排盲评视频：本方法 vs 基线（**无动作 / 随机轨迹回放 / 规则状态机（即 reachy-mini-demo 小艺）/ 离散 token 版**），评自然度/适切性/"活着感"，20–30 名评分者。

- 消融阵容：语义通道形式（隐状态 vs 回复文本 Plan B）、-本体反馈、回归 vs flow matching、冻结 vs 联合微调（v3）、块长/重叠超参。

### 5.4 基准打包

金标准测试集 + 指标代码 + VLM 裁判协议 + 全部基线实现 = 命名基准（名字待定，见 §8），随论文开源。

**协作循环协议**：每阶段末，服务器端将实验报告（结果+代码+踩坑，中文）提交进 GitHub 仓库；本地拉取审查、修航向、发出下一阶段任务书。服务器端遇本文档未覆盖的设计决策时：小事自决并在报告中声明，大事（改动作表示/数据 schema/评测协议）必须停下写入"待决问题"等本地裁决。

**降级预案**：M2 末回归版仍不收敛 → 砍聆听动作、只做说话动作（范围减半，"边说边动"故事仍立）；M3 末 flow matching 在 NPU 上不可行 → 回归版+多样性正则做主结果，flow matching 移 GPU 云机做小规模对照。

## 7. 风险清单（按危险度排序）

1. **隐状态获取**（v1 语义通道，M1 首周必验）：训练期用**离线批量预计算**（transformers 慢路径跑一遍数据集并缓存，不碰 vLLM）；M2 仿真 demo 用 transformers 路径 1Hz 在线前向（910B 上可行）；真机 demo 前再解决 vLLM 在线导出。**双保险**：若首周离线路径受阻，立即切换回复文本 Plan B（§3.3 表，已完整设计，接口不变），不许在此卡壳超过一周。
2. **NPU 算子兼容**（flash-attn 变体、DiT、StarVLA 组件）：M1 首周冒烟测试，问题早暴露早绕行。
3. **8B 微调链路未验证**（仅影响 v3"B 收尾"）：昇腾上尚未跑通过 JoyAI/QwenVL 的微调；候选 LLaMA-Factory / ms-swift。M3 末预研，不通则 v3 直接放弃（主线不受影响，消融表少一行）。
4. **评测循环论证**：已由 §4.4 防泄漏设计消解。
5. **时间**：DyaPlex 说明方向升温，11 月是硬墙；一切"可选项"（第 3 层、v3、ICLR）都不得挤占主线。

## 附录 A：VLA 速成路径（按"对本项目有用"排序）

1. **ACT / ALOHA**（action chunking 的出处）——只看 chunking + temporal ensembling 两节，理解 §2 洞察 1 的桥接原理。
2. **Diffusion Policy**——理解"动作分布多峰→生成式头"的动机，对应 v2。
3. **π0**（Physical Intelligence）——flow matching 动作头 + VLM 骨干的标准组合，v2 的直接参照。
4. **GR00T N1 / Figure Helix 博客**——"大 VLM 慢脑 + 小专家快脑"双系统，本架构的 manipulation 界同构。
5. **OpenVLA**——离散 token 路线代表，理解我们 baseline（方案 A）在做什么。
6. **DyaPlex（arXiv:2606.03874）+ LiveGesture（arXiv:2604.10927）**——最近邻，读法：他们缺的四要素我们怎么补。
7. **代码**：StarVLA（模块拆法）、LeRobot（数据格式）。LIBERO 数据用 h5py 打开看一遍即可祛魅：HDF5 = 二进制数组容器，每段演示存图像序列+本体状态+7 维动作@20Hz，与"图片+文本 JSON"没有本质区别。
