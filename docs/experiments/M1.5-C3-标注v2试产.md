# M1.5-C3 标注任务 v2 试产报告（含成本告警，**停下等审查**）

- 日期：2026-07-16
- 执行：本机侧（Windows 11）
- 依据：M1.5 计划 C3；契约 C0 修正案（action_label 值域扩为 85 情绪动作∪5类∪none）
- 状态：**试产已跑但中途主动止血**。发现**成本会失控**，需用户裁定省钱策略后再续。**未全量。**

## 1. 做了什么

- 建 `annotation/.venv`（openai+ijson，清华镜像），装好可跑。
- **改造 prompt 为 v2**（已 commit `9446ca4`）：标签集 5 类 → 从 `data_gen/motion_library/index.json`
  **程序化提取的 85 个情绪动作**（kind==emotion）+ none；舞蹈库排除；dance1/2/3 仅音乐/舞蹈语境。
  parse 按 86 标签校验（最长名优先防子串）。4 个无描述补录动作补了最简英文注释。
- API 链路验证：10 条烟测 10/10 ok；60 条 chat 分布 85% none + 合理命中。
- **试产**：从 background/chat/narration/event_grounding 各取 ~2500 样本（narration 止血于 500）。
  实际产生 **15,900 次 API 调用**、标注约 1.4 万条 response。

## 2. ⚠️ 头号发现：按当前设计，全量成本会失控

网关实测计费（gemini-2.5-flash @ 一步/yibuapi，用户提供价）：输入 \$2/1M、输出 \$5/1M、缓存读 \$0.2/1M。

| 指标 | 实测 |
| --- | --- |
| 平均 prompt_tokens/次 | **2466**（其中 85 词条 catalog ≈2400） |
| 平均 completion_tokens/次 | 6.7（关思考生效，reasoning=0） |
| 命中缓存 | **仅 0.57%**（20 并发下 implicit cache 基本没命中，每次全额付 catalog） |
| **本次试产实花** | **\$78.95**（3920 万输入 token） |

**根因**：85 词条的 catalog（~2400 token）作为 system prompt **每条 response 都全额重发一次**，
且网关几乎不缓存。按每条独立一次调用外推**全量 1000 万 response**：

| 方案 | 估算全量成本 |
| --- | --- |
| 现状（每条 1 次调用，无缓存） | **≈ \$49,600** |
| 若 catalog 全走缓存（\$0.2/1M） | ≈ \$7,900 |
| **批处理**：每次调用塞 N 条 response 摊薄 catalog（N=50） | ≈ \$2,280 |
| 批处理 N=50 + **只标 chat（真对话）**、其余默认 none 不调 API | **数百～\$1000 量级**（取决 chat 占比） |

> 理论地板：每条 response 的文本（context+reply ≈66 token）至少发一次 → 1000 万条 ×66×\$2/1M ≈ **\$1,320**。
> 要压到「几百美元」，**必须**：①批处理摊薄 catalog；②缩小标注范围（只标真对话，或标子集）。

## 4. 质量观察（关键，牵动范围决策）

- **chat（真对话）质量可用**：确认→understanding2/yes1、完成→success1、兴奋→enthusiastic1，站得住。
  唯一偏差是「肯定性事实确认」倾向贴 understanding2 而非 none，属可调口径。
- **narration / event_grounding 明显过度赋情绪**：这两类是**第三人称视频描述/事件播报**，不是机器人自己的话。
  例：narration「May 急切地为未婚夫辩解」→ anxiety1、「火药味越来越浓」→ rage1——模型被**被描述内容的情绪**
  带偏，而机器人只是在**旁白**，本不该做该身体动作。event_grounding 仅 45.7% none 同理虚高。
  → 建议：**这两类（连同 background）默认 none、不调 API**，既省钱又提质；情绪动作只标 `chat`。

## 5. LLM 调用健康度

15,900 次调用 **finish_reason 全 = stop、0 空返回、0 error**。关思考(budget=0)+MAX_TOKENS=512 的机制完好，
**无「思考 token 吃光→静默误标 none」现象**。即：机制没问题，问题在成本与范围。

## 7. 结论与待决策（**停在此，等用户 + 本地 Fable5 审查**）

- ✅ 机制健康、chat 标注质量可用。
- ❌ 按「每条 response 全额重发 catalog」的现状，全量 ≈ \$5万，**远超「几百」预算**。
- 待用户裁定（大事，涉及成本/范围/调用机制）：
  1. **范围**：是否只标 `chat`（真对话），把 narration/event_grounding/background 一律默认 none 不调 API？
     （推荐：是——省钱且避免旁白被误贴情绪）
  2. **机制**：是否改为**批处理**（每次调用塞 N 条 response，摊薄 85 词条 catalog）？这是把成本从万级压到
     千/百级的关键手段，但需改脚本（输出改 JSON 数组、失败按批重试）。
  3. **规模**：全量 ~1000 万，还是先标一个子集（如几百万）把总价钉死在几百美元？

本次已花 \$79。裁定前**不再花钱**。
