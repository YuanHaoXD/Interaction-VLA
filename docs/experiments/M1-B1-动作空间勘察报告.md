# M1-B1 动作空间勘察报告

> 任务：动作空间勘察——9 维安全限位实测 + 30Hz set_target 压测
> 环境：本机 Windows 11，仿真 daemon（`reachy-mini-daemon --sim --scene minimal`），reachy-mini 1.8.0 + MuJoCo 3.3.0
> 执行：2026-07-14，B 系列本机侧
> 产出：`sim/probe_action_space.py`、`sim/limits.json`

## 1. 结论速览

- ✅ **30Hz 达成**：`achieved_hz = 30.0003 Hz`，`max_jitter_ms = 4.61 ms`（远低于 10ms 阈值）。set_target 流式控制在本机仿真下**完全胜任 30Hz 实时驱动**，块长/频率无需下调。
- ✅ 9 维安全范围全部产出（body_yaw 按设计取文档保守值，待 A4 核实）。
- ✅ 头部平移/转头、天线大幅摆动均由用户肉眼确认**正常达位、无抖动无穿模**。
- ⚠️ 三处需记录的观察：①`ReachyMini()` 连接耗时不稳定；②天线维度未测到真实触顶（±3.2 为扫描上限）；③仿真安全范围显著大于真机 CALIBRATION 值。

## 3. 30Hz 压测结果

```
30Hz 压测: {'achieved_hz': 30.000258856515085, 'max_jitter_ms': 4.6050666673788}
```

- 方法：pitch 正弦（freq=1Hz、amp=0.17rad）连续 30 秒 @30Hz 流式 set_target，忙等（busy-wait）对齐时间片。
- **达成频率 30.0003 Hz**：忙等策略下几乎零漂移。
- **最大周期抖动 4.6 ms**：单帧最坏情况 33.3+4.6 = 37.9ms，仍在可接受范围。
- 意义：动作块（1.5s @30Hz = 45 步）的实时回放**不受频率瓶颈限制**，后续 B3 回放器、M2 实时驱动均可放心用 30Hz。

## 4. 关键异常与观察（勘察的核心价值）

### 4.1 `ReachyMini()` 连接耗时不稳定（重要，影响后续所有 SDK 脚本）

首次跑本脚本时**静止 >20 分钟无产出**，一度疑似卡死。诊断（`sim/_diag_sdk_calls.py`、`sim/_diag_sweep.py`）定位根因：

- **卡点在 `ReachyMini()` 构造/连接握手**，而非任何控制调用。同一 daemon 下实测构造耗时波动 **10.3s ~ 52s**，首次冷连接疑似更久。原因是 SDK 连接自动检测（探测 USB/localhost/网络，见 [Seeed 开发指南](https://wiki.seeedstudio.com/reachymini_agents/)）。
- **一旦连上，调用极快（本地内存级）**：`set_target` 平均 **0.18 ms/call**（理论 5696Hz）、`get_current_head_pose` ~0ms。30Hz 毫无压力。
- **叠加块缓冲坑**：Python `print` 经文件重定向是块缓冲，构造期日志迟迟不 flush，加剧"卡死"假象。**排障对策**：跑 SDK 脚本一律用 `python -u`（无缓冲）；判断死活看①MuJoCo 窗口机器人是否动、②probe 进程 CPU 时间是否增长。

> 已存本机记忆 `reachy-mini-sim-connect-slow` 供后续任务复用。

### 4.2 仿真安全范围显著大于真机

| 维度 | 本机仿真实测 | 真机 CALIBRATION.md 值 | 差异 |
|---|---|---|---|
| pitch | -43°/+40° | +15°/-10°（手势用） | 仿真约 3~4 倍 |
| yaw | -69°/+52° | ±15° | 仿真约 3~4 倍 |
| roll | ±46° | 15° | 仿真约 3 倍 |
| 天线 | ±183°（未触顶） | ±0.8rad ≈ ±46° | 仿真远超 |

原因：**仿真无硬件限位、无 Dynamixel 过载保护、跟踪精度高**（误差恒小于容差，直到扫描上限）。真机受电机物理限位与过载锁存约束（CALIBRATION 记有左天线 Overload Error 锁存需断电清除）。

**结论与建议**：B2 模板库、后续安全层若直接吃 `limits.json` 的仿真值，幅度会远超真机可执行范围。**建议**：数据生成（B2~B4）阶段用仿真值无妨（回放在仿真里）；但**真机部署（M3）前必须用真机 CALIBRATION 的保守值覆盖 `safe_min/safe_max`**，否则真机会撞限位/触发过载。此点交叉引用 A4 的 SDK 硬限位调研。

### 4.3 天线未测到真实触顶

天线 safe range = ±3.2 恰等于脚本 `max_abs=3.2`，且 `err_tol=0.15` 全程未超 → **扫描在到达上限前从未检测到触顶**，±3.2 是**扫描量程上限而非真实限位**。用户肉眼确认天线大幅摆动正常。真实仿真天线上限未知（可能 >3.2）。真机限位以 A4/CALIBRATION 为准。

## 5. max_vel 修订

脚本写出的 `max_vel` 为初值 `[0.1,0.1,0.1, 2.0,2.0,2.0, 2.0, 6.0,6.0]`（m/s 或 rad/s）。B1 压测未直接测量各维最大安全角速度（压测只测循环频率），故**本阶段暂不修订 max_vel，沿用初值**。这些值将在 B2 `compose` 的逐步限速中作为连续性约束使用；若 B4 回放观感出现"太快/不跟手"，再回头据实测修订。

> 待办（非阻塞）：如需精确 max_vel，可另写单维阶跃响应测试测各维实际跟踪速度上限。当前初值对数据生成足够保守。

## 6. 待决问题

- **无 B 侧独立大事待决**。仿真 vs 真机限位差异（§4.2）不是设计变更，是如实记录的勘察事实，处理方案（真机前覆盖限位）已给出，交 A4 核实硬限位。
- body_yaw 实测缺失（SDK 无可靠 body_yaw 读数，Stewart 平台反向补偿使头部世界姿态读数不反映 body 转角），取文档保守值 ±2.8，**待 A4 核实**。
