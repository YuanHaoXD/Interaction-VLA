- 受众：**Fable5（本机侧设计/审查 AI）** —— 请据此裁定 demeanor 方向是否成立，并**构造更细的动作分类准则**
- 相关：先行报告 `M1.5-C3-标注v2试产.md`（emotion 模式 + 成本告警）；本文件是其**策略转向后的续篇**
- 一句话：emotion 模式（每条判情绪、可 none）质量/成本都不理想 → 用户改判 **demeanor 模式**（判"人说这句话该有的神态"、**去掉 none**）→ 试标质量明显变好，但**85 个动作里近义动作太多，模型分不清就塌到 understanding1（47%）**，需要 Fable5 给出可判定的细分准则。

---

## 1. 起因：为什么从 emotion 转 demeanor，为什么现在要新准则

### 1.1 C3 任务背景
标注 v2 的目标：给 JoyAI 数据集每条 `response` 打一个**官方动作库动作名**（85 情绪动作，词表从 `data_gen/motion_library/index.json` 程序化提取），作为交互 VLA 数据第 1 层的动作语义锚（设计文档 §4.1）。

### 1.2 emotion 模式（旧 v2）的两个问题（详见 `M1.5-C3-标注v2试产.md`）
1. **过度赋情绪 + 语义错位**：prompt 让模型"判断 response 本身表达的情绪"。结果 narration / event_grounding（**第三人称视频描述/播报**）被**被描述内容的情绪**带偏——例："火药味越来越浓"→`rage1`、"May 急切辩解"→`anxiety1`。机器人只是在旁白，不该做这些身体动作。
2. **成本失控**：85 词条 catalog（~2400 token）每条 response 全额重发、网关几乎不缓存（实测 0.57%）。按网关价（输入 \$2/1M）全量 1000 万 response ≈ **\$4.9 万**。本次已花 \$79。

### 1.3 用户 2026-07-16 的策略转向（本文件试的就是这个）
> "提示词别让模型判断回答的情绪，而是判断**一个人在说这句话的时候应该是什么神态**；然后**把 none 去掉**。因为正常人说话，就算是客观描述也会有表情动作神态。"

即：从"这句话表达了什么情绪 → 否则 none"，改为"**一个人自然说出这句话时会带什么神态 → 必选一个**"。

## 2. demeanor 试标：做了什么 + 得到什么

### 2.1 实现（已改 `annotation/annotate_actions_api.py`，作为可回退开关**已提交**为中间态，等本裁定后按新准则重写 prompt）
- 新增 `LABEL_MODE` 开关：`demeanor`（默认，新）/ `emotion`（旧 v2，可回退）。
- demeanor 模式：**候选 = 85 情绪动作，无 none**；prompt 改为"判断说这句话的自然神态，必选其一"；并加规则 **"平淡/客观陈述优先选温和日常神态（attentive1/understanding1/thoughtful1…），强烈神态（rage1/surprised1…）留给确实值得的句子"** 来抑制 1.2 的带偏。
- 解析失败/委派用**可见哨兵**（`__unparsed__`/`__delegation__`），不再静默塞 none。

### 2.2 规模与健康度
- 试标 455 条 response（chat 100 样本 + narration 20 样本≈255 条 + event_grounding 100）。
- **健康度满分**：finish_reason 全 = stop，0 空返回、0 解析失败、0 委派。
- 成本 ≈ \$2（会话累计 ≈ \$81）。

## 3. 暴露的核心问题（**这就是要 Fable5 解决的**）

**85 个动作里近义动作太多，模型分不清就塌到少数几个**：understanding1 + thoughtful1 + attentive1 三者吃掉 **77.6%**。用户原话：*"讲解的时候点头动作也有很多个"* —— 光"讲解/肯定"就有 understanding1、understanding2、yes1、attentive1… 好几个近义动作，缺乏可判定的分界，模型只会都投给 understanding1。

**证据：三个主导标签各自吸收了本应细分的子类型**

`understanding1`（47%，全是"讲解型复述/报时长/客观陈述"，其实混了好几种语气）：
- "Yes, he finished. The man raised his right arm…"（**肯定确认** ← 更像 yes1/understanding2）
- "The full 360-degree rotation lasted about 4 seconds…"（**中性报数据** ← 也许该有更"平铺直叙"的动作）
- "The quad bike did a burnout with smoke billowing…"（**客观复述事件** ← 讲解貌）

`attentive1`（14%，全是"引导观众注意/开场介绍"）：
- "好，画面一开始是新闻主播…看来今天要关注半导体消息"（**开场引入**）
- "注意看，画面右侧出现了背着十字架的男子，这是最核心画面"（**提示重点** ← "注意力引导"确实贴 attentive，但和 understanding1 边界靠语气）

`thoughtful1`（16%，混了"空镜氛围描述"和"真·若有所思"）：
- "镜头切到野外，灰绿灌木和紫色小花，空旷静谧"（**氛围空镜** ← 未必是"找点子"的 thoughtful）
- "他一边走一边低头看手机…似乎在确认路线"（**推测/沉吟** ← 更贴 thoughtful1 本义）

→ 结论：**分类塌陷不是模型坏，是候选集内近义动作缺乏"可操作的分界准则"。** 需要 Fable5 定义：在"讲解/肯定/聆听簇"内，用什么可判定特征（句子功能？语气？是否确认？是否提示重点？）把它们分开。

## 4. 支撑材料：85 动作近义簇地图（帮 Fable5 定分界）

> 完整 name+description 见 `data_gen/motion_library/index.md`；下面按语义聚簇，**标注易混簇**。每簇内动作缺乏天然分界，是分类准则要处理的重点。

- **🅰 肯定/理解/点头（★塌陷重灾区）**：`understanding1`(点头表示听懂)、`understanding2`(点头且同意/亦作yes)、`yes1`(长肯定/确认)、`yes_sad1`(忧郁的yes)、`proud2`(满意/亦作yes)、`resigned1`(无奈的yes/勉强OK)
- **🅱 聆听/专注/引导注意**：`attentive1`(表示在听、鼓励继续)、`attentive2`(听人说话时)、`curious1`(环视多人)
- **🅲 思考/找点子/迟疑/发问**：`thoughtful1`(仰头找新点子)、`thoughtful2`(抬头想点子)、`inquiring1/2/3`(追问/思索发问)、`uncertain1`(没主见)、`confused1`/`lost1`(不知如何答)、`incomprehensible2`(没听懂)
- **🅳 开心/兴奋/庆祝/成功**：`cheerful1`、`enthusiastic1`(惊人喜讯)/`enthusiastic2`(普通好消息)、`laughing1/2`、`success1`(完成任务)/`success2`(庆祝)、`proud1`(满意环顾)/`proud3`(我做到了)、`dance1/2/3`(舞蹈，**默认排除，仅音乐/舞蹈语境**)
- **🅴 惊讶/害怕**：`amazed1`、`surprised1/2`、`fear1`、`scared1`、`anxiety1`、`electric1`(通电)
- **🅵 否定/不满/生气（19 个，粒度极细）**：`no1`/`no_excited1`/`no_sad1`、`displeased1/2`、`contempt1`、`disgusted1`、`irritated1/2`、`frustrated1`、`furious1`、`rage1`、`reprimand1/2/3`、`go_away1`、`impatient1/2`、`indifferent1`
- **🅶 悲伤/尴尬/出糗**：`sad1/2`、`downcast1`、`lonely1`、`shy1`、`uncomfortable1`、`oops1/2`、`dying1`(装死)
- **🅷 无聊/疲惫/睡（多为长时系统动画，对话罕用）**：`boredom1/2`、`exhausted1`、`sleep1`、`tired1`、`waiting`、`mini-deep-sleep`
- **🅸 社交/招呼/安抚/帮助/感谢**：`welcoming1/2`、`come1`、`calming1`、`helpful1`(乐于帮忙)/`helpful2`(道谢)、`grateful1`、`loving1`、`relief1/2`、`serenity1`、`wake-mini-up`、`toc-toc-toc`

> 4 个无描述的补录动作（`waiting`/`mini-deep-sleep`/`wake-mini-up`/`toc-toc-toc`）是系统动画，语义弱，**建议考虑从对话候选表剔除**。

## 5. 请 Fable5 裁定的问题

1. **方向**：demeanor（判神态、无 none、必选一个）是否作为 C3 定稿方向？（本机试标观感：比 emotion 模式明显合理）
2. **★分类准则（核心）**：如何在易混簇（尤其 🅰肯定/理解 与 🅱聆听 与 🅲思考）内给出**可判定的分界**，避免全塌到 understanding1？建议产出一张"决策准则表"——每个候选动作配 1 条**可操作的触发条件 + 反例**（例：understanding2 用于"确认+同意"、yes1 用于"明确回答是/长肯定"、attentive1 用于"提示对方/观众注意、引导继续"、thoughtful1 仅用于"真的在沉吟/推测"而非中性空镜）。这张表会直接变成新的 prompt 词条。
3. **候选表裁剪**：是否把**对话中几乎用不到的动作**（🅷无聊/睡的长动画、`electric1`/`dying1` 等系统/整活动画、4 个无描述补录动作、🅵里过细的 19 类否定）**移出候选表**？裁剪既提质（少近义干扰）又降成本（catalog 变短）。
4. **成本机制**（可与 3 合并考虑）：去 none 后每条都要真调 API，全量仍是 catalog 每次重发的老问题（~\$5 万量级）。**批处理**（每次调用塞 N 条 response 摊薄 catalog）是压到千/百级的关键，但需改脚本（输出改 JSON 数组、按批重试）。用户暂缓、待想清楚——Fable5 可一并给意见（候选表越短、批处理越省）。
