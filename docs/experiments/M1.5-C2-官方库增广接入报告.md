## 2. 接口（C4 依赖，签名冻结）

```python
load_lib_move(name: str) -> np.ndarray[T,9]           # float32, 均匀 50Hz
augment_lib_move(traj, rng, time_scale=(0.85,1.15), amp_scale=(0.7,1.15)) -> np.ndarray[T',9]
# compose() 对 label.startswith("lib:") 走库路径:augment_lib_move(load_lib_move(name), rng, **ev.get("aug") or {})
```

实现要点：
- `load_lib_move`：调 `build_motion_library.load_saved_move(name, fps=50)` 取 action9。**重采样到均匀 50Hz** 而非原生返回——4 个动作（mini-deep-sleep / toc-toc-toc / waiting / wake-mini-up）原生含重复时间戳，均匀化消除毛刺，也保证 compose 的等步长假设成立。
- `augment_lib_move`：①时间缩放 k∈time_scale 线性重采样到 `round(T*k)`；②幅度缩放取**头部姿态(0-5)/身体(6)/天线(7-8)三组独立系数**（皆来自 amp_scale，满足计划“天线/身体与头部可用不同系数范围”）；③首尾各 0.2s 乘 smoothstep 压到 0；④裁剪到 `limits_real`。函数**不改入参**（`test_augment_does_not_mutate_input` 守护）。
- `compose` 用**懒加载** `from data_gen.augment import ...` 规避 templates↔augment 循环导入。`lib:` 事件不做 4s 截断（保留完整情绪动作），仅在超出 episode 末尾时自然截断。

## 3. limits_real.json —— 真机可行域（自决点：max_vel 取法）

扫全部 104 个官方动作 action9（均匀 50Hz），逐维取 min/max、最大逐步速度：
- **位置**：`safe_min/max` = 逐维 [min,max] 各向外扩 range 的 10%（计划：“加 10% 裕量”）。位置包络多数维**比仿真 `limits.json` 更紧**（如 roll ±0.62 vs 仿真 ±0.80、pitch [-0.49,0.63] vs 仿真 [-0.75,0.70]），印证 B1 §4.2 的仿真虚高。
- **max_vel（自决并声明）**：取库内逐维**全局最大**逐步速度 ×1.1，而非分位数。理由：库动作录制于真机→真机确能达该速度；若 max_vel 设在库内极值之下，`compose` 的逐步限速会**扭曲**直接插入的库动作。库动作段的连续性改由 augment 的首尾 smoothstep 保证，`compose` 的限速器退化为 NaN/尖峰兜底。极值多来自舞蹈（grid_snap 等，设计默认排除）与 wake-mini-up，故 vmax 偏大（pitch 35、天线 47）——这是“不裁剪任何真机动作”的代价，可接受。

## 4. 生产端限幅切换的影响（自决并声明）

按计划“限幅统一改用 `limits_real.json`”，`make_samples`/`pipeline` 的 `LIMITS` 由 `sim/limits.json` 换为 `sim/limits_real.json`（compose 签名不变，仍收 `limits` 入参，仅调用方改传——不触碰 C4 依赖）。
- 重生成 100 段模板样例：**仅 33 段 actions 变化**（多数模板幅度本就在两套限位内；被改的 33 段是此前被仿真 `max_vel=2.0` 裁过速度、或被更紧仿真位置限位裁过的段——如 nod 峰速 ~3.3rad/s 曾被限到 2.0，现放开）。`meta.json` 零变化，timeline 零变化。
- **隐状态 fixture 保护升级**：`make_samples` 新增 `_merge_prev_meta`，重生成时自动保留已存在 meta 中的外部键（`d_model`/`n_decisions`）。ep_12/33/58 三段隐状态 fixture 的键与 `hidden_states.npy` 均完整（13/17/16 决策，形状 (n,4096)）。此后重生成不再需要手工恢复。

> C1 的连续性单测 `test_compose_respects_limits_and_continuity` 仍显式传 `sim/limits.json` 并按其 max_vel 断言——测的是 compose 机制而非生产配置，故不受影响，保持绿。
