# C3 标注裁定：簇级 demeanor + 候选裁剪 + 成本方案（Fable 5，2026-07-16）

> 回应 `docs/experiments/M1.5-C3-demeanor试标与分类准则待裁.md` 的四个待裁问题。
> 本裁定生效后，C3 按 §5 的执行步骤继续；涉及 A5 标注契约的修正见 §4。

## 0. 裁定总览

| 问题 | 裁定 |
|---|---|
| 1. demeanor 方向 | ✅ 成立（判"说这句话的自然神态"、无 none、必选一个） |
| 2. 分类准则 | ⚠️ **不做 85 选 1**。改为**两级体系**：LLM 只判 **13 个神态簇**（准则表见 §2）；簇内具体动作由数据生成时**按权重随机采样**（§3） |
| 3. 候选裁剪 | ✅ 裁。17 个系统/整活/长时动画移出对话词表（清单见 §2 末） |

## 1. 为什么塌陷不可能靠"更细的准则"修好

understanding1 / understanding2 / yes1 的差别在**轨迹形态**（点头的幅度、次数、余韵），不在**语句内容**。同一句"对，他完成了"配这三个动作都成立——**文本→具体动作本来就是多峰分布**，让 LLM 强行 argmax 就是在制造噪声标签。而"这句话是在肯定、还是在讲解、还是在思考"是文本可判定的。所以：**LLM 判到它能判的层级（簇），判不了的层级交给随机性（簇内采样）**。副产物：簇内采样天然保证了近义动作全被用到，多样性由构造保证——这与我们对沉默段多峰的处理哲学完全一致。

## 2. 神态簇准则表（13 簇，直接变成 prompt 词条）

> 每簇 = 触发条件 + 反例。LLM 输出**簇名**。簇内成员与权重见 §3 映射表。

| 簇名 | 触发条件（句子的功能/语气） | 反例（别选它） |
|---|---|---|
| `affirm` 肯定认同 | 核心是"对/是的/没错/同意"，明确表态认可 | 中性复述画面 → explain |
| `explain` 讲解陈述 | 中性说明、报数据、复述所见，无明显情绪起伏 | 带"注意看/我们来看"引导 → attend；带"似乎/可能"推测 → think |
| `attend` 引导关注 | 提醒注意某处、开场引入、鼓励对方继续说 | 纯中性陈述 → explain |
| `think` 思考推测 | 沉吟/推断/自问："可能、似乎、我想想、为什么呢" | 答不上来 → unsure；中性空镜描述 → explain |
| `unsure` 困惑不解 | 没看懂/答不上/信息不足 | 有方向的推测 → think |
| `joy` 开心兴奋 | 好消息、精彩瞬间、夸赞、成功达成 | 中性的"完成了" → explain；轻微满意 → affirm |
| `surprise` 惊讶 | 意外转折、反差、"竟然/没想到" | 惊喜且高兴 → joy；惊吓 → fear |
| `fear` 紧张担忧 | 危险将至、替人担心、紧张不安 | 坏结果已发生的难过 → sad |
| `negate` 否定拒绝 | 核心是"不/不是/不行/不同意"，语气平静 | 带怒气的驳斥 → annoy |
| `annoy` 不满恼火 | 被冒犯、批评指责、强烈不满 | 平静的不同意 → negate |
| `sad` 悲伤低落 | 难过、遗憾、离别、共情对方的痛 | 替人紧张 → fear |
| `warm` 温暖安抚 | 问候、感谢、安慰、欢迎、表达好感 | 单纯高兴 → joy |
| `awkward` 尴尬歉意 | 自己出错、道歉、不好意思 | 批评别人的错 → annoy |

**移出标注词表（17 个）**：`boredom1/2`、`sleep1`、`tired1`、`exhausted1`、`waiting`、`mini-deep-sleep`、`wake-mini-up`、`toc-toc-toc`、`electric1`、`dying1`、`dance1/2/3`、`go_away1`、`impatient1/2`、`indifferent1`。舞蹈类留音乐语境，系统动画（`mini-deep-sleep`/`wake-mini-up`/`toc-toc-toc`/`electric1`/`dying1`）不入训练。

### 2.1 补充裁定（2026-07-16，用户提议）：待机类升格为"长沉默事件"机制

`waiting`/`boredom1/2`/`tired1`/`sleep1` **不进标注词表，但进数据生成**——官方描述本身就定义了待机行为链（"没人理你 2 分钟→boredom1→boredom2→sleep1"），做成**时长触发的沉默事件注入**（属 C4 的 `convert_sample`/`compose` 层，不涉及标注）：

**三条规则（2026-07-16 与用户确认的定稿）**：

1. **底座 = 呼吸感，永远在**：程序化 idle（z 呼吸 + 头部慢漂，B2 已实现）是系统默认态，任何没动作的时刻都是它。
2. **点缀 = 稀疏注入 + 冷却**：沉默间隙内注入待机动作，**冷却 ≥20s**（插完必回纯呼吸，绝不连轴转）；每次注入过增广（时间/幅度抖动），同一动作两次出现不重样。
3. **困意随间隙时长升级**（分层池，权重为初值）：
   - 15~30s：`serenity1`(0.35)、`thoughtful2`(0.25)、`waiting`(0.25，**入池前需仿真肉眼验收**——官方无描述)、`tired1`(0.15)
   - 30~60s：上池 ∪ `curious1`(0.25)、`lonely1`(0.25)、`tired1` 权重升
   - >60s：`boredom1`(0.5) → 同一间隙再注入时升级 `boredom2`(链式)
   - 不入池：`exhausted1`/`sleep1`/`mini-deep-sleep`（18~20s 太长，M2 观感后再议）；`relief/calming` 类是对人交互动作，非待机。
- 注入走 `lib:` 路径（增广+边界连续化），随机种子入 meta，池与权重放 `data_gen/cluster_map.json` 的 `_idle_pool` 键（同文件版本化）。
- 训练价值：模型从隐状态学到"长时间没动静 → 困倦行为"，部署时"放着不管会打瞌睡"是活物感的招牌 demo。真实阈值（官方 2 分钟）与数据阈值（15/30/60s）差异属 episode 长度上限导致的尺度压缩，接受。

## 3. 簇→动作映射与采样（数据生成侧）

- 新建 `data_gen/cluster_map.json`（入 git，版本化）：`{簇名: [{move, weight}]}`。初版成员按报告 §4 的簇地图对号入座（执行者按 `index.md` 描述核对微调，属小事，报告里声明即可）；权重初值：日常动作高（如 understanding1/attentive1/enthusiastic2），强烈动作低（furious1/rage1 只配 annoy 簇的极端句，权重 ≤0.05）。
- `pipeline.convert_sample`：annotation 的 `action` 字段现在是**簇名** → 按 cluster_map 加权采样出具体动作 → 生成 `lib:<动作名>` 事件（随机种子入 meta，可复现）。
- **动作密度控制（新增，重要）**：不是每条 response 都渲染事件动作——每簇配一个**渲染概率**（初值：explain 0.35、attend/affirm/think 0.6、其余 0.85），未渲染的 response 只保留 idle+backchannel，簇标签仍留在 timeline 里（信息不丢）。理由：真人不会每句话都做大动作，全渲染会训出"戏精"机器人；密度参数留给 M2 观感调优。

## 4. A5 标注契约修正（随本裁定生效）

- annotation 产物的 `action` 字段值域：**13 个簇名**（替代此前"85 动作名 ∪ 旧5类 ∪ none"）。
- `data_gen/example_annotation.json` fixture 与 `pipeline` 测试同步更新（C4 侧改动，服务器做）。
- timeline 事件的 `action_label` 仍是具体 `lib:<动作名>`（采样后），episode 侧 schema 不变。

## 5. C3 执行步骤（据此继续）

1. prompt 重写为 13 簇选择题（§2 表格逐行进 prompt；输出簇名；哨兵机制保留）。
2. **批处理改造**：每次调用打包 N=20 条 response（编号列表进、JSON 数组出、逐条校验、失败单条重试、重试上限 3）。
3. 小试 500 条（约 \$1）→ 看簇分布与抽样质量，塌陷阈值：单簇 >40% 即回报再裁。
4. 通过后试产 1 万条 → 报告（分布+50 条抽样）→ 用户+Fable 5 审查 → 按 M2 需求量全量（**首期目标 = 覆盖 10 万段所需的 response，约 50 万条，不是 1000 万**；预算上限 \$200，超了停下来报）。

## 6. 成本账（供核对）

簇级 catalog ≈ 700 token（含准则表+指令）；批 20 条摊薄后每条输入 ≈ 120 token、输出 ≈ 10 token。50 万条 ≈ 6500 万 token 输入 ≈ \$130 + 输出零头。比原方案（\$4.9 万）降约 250 倍，其中：词表裁剪+簇化 5×、批处理 20×、标注量对齐实际需求 2.5×。
